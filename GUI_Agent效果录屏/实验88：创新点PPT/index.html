Model Foundation

直觉建模：N 叉树地图

虽然网站本质上是复杂的图（Graph），但在交互层级上，N 叉树（N-ary Tree） 更符合人类对交互路径的直觉认知。

现状：Agent 的“直觉摸索”困局

行动机制：分支路口的反复观察

目前的 Agent 在每一个分支路口，都必须停下，进行 OCR 识别、DOM 分析和推理，才能跨出下一步。

Pain Point 01

推理时延 (Latency)

每前进一步都要调用一次大模型。在高频业务中，这意味着无法接受的交互成本。

Pain Point 02

概率依赖 (Probability)

它在每一个 Step 都在做概率预测。一旦一个环节出错，整个链路即刻崩溃，容易迷路陷入死循环。

比喻：初访异乡的游客

现在的 Agent 就像一个初到异国他乡的游客。他能看懂路标（识别按钮），但他脑子里没有这张城市的全局地图。

“每走到一个十字路口，都要分析 A、B、C 猜测哪个是火车站。”

“他可能走进了死胡同，或者在原地绕圈。”

“最核心的问题：下一次去火车站，他还得重新分析一遍，因为他没有固化路径。”

核心演进：先测绘，后导航

构建地图

我们的任务不是为了从 A 点到 B 点，而是为了跑遍全城的每一条路，形成拓扑底座。

RPA 导航

RPA 是严格执行指令的自动驾驶车。它不需要理解“为什么要转弯”，它只需接收序列指令：

Drive(500m) -> TurnLeft -> Stop
跳过思考与分析路标的时间，直接全速前进。
01

构建地图：专用智能体 (Cartographer)

开发一个“测绘绘图员”。它不被分配具体任务，只负责在目标网站进行全量遍历。

节点持久化

所有的节点、按钮、输入框及其关联关系全部存储在 Neo4j 图数据库 中。

DSL 语义定义

使用领域特定语言描述状态转移逻辑：

(State A) --[Action(input/click)]--> (State B)
02

条件路径与人工纠偏

如何处理“动态路径”？

例如：只有输入了正确的订单号，才会显示“退款”按钮。

属性增强（Pre-condition）：

Edge: { needs: "order_id" }

人机协同标注

当 AI 自动探索失败时，自动通知标注员。标注员的一次点击即是对地图的实时纠偏与经验固化。

构建阶段允许出错，
执行阶段绝不允许出错。

组件分析：Cartographer 绘图员

绘图员本质是在对网站的 业务逻辑状态机 进行逆向工程。

规避“死循环”陷阱

盲目遍历会导致陷入“万年历”或“复杂验证码”等死循环组件。我们的 AI 绘图员具备语义识别能力，会优先点击有意义的业务元素，规避无效探索。

组件分析：Navigator 导航员

01

语义对齐意图理解

“我买的鼠标不好用，退钱。”

Agent 识别到“退钱”即是地图中的“申请售后”节点，实现模糊指令到标准化路径的映射。

02

图路径寻找

在 Neo4j 中查找从首页到下载页的最短语义合规路径。

03

RPA 全速执行

将路径一次性转化为高效的 Python 代码：

DrissionPage.click("@text=申请售后")

Strategic Significance

数字化资产沉淀

当我们拥有了数百个业务系统的完整地图时，这本身就是巨大的环境知识库。 它是实现跨业务系统流程自动化、打破数据孤岛的关键基础设施。

模型能力的增强补丁

网站地图相当于为通用大模型注入了领域专业先验。地图本身就是模型能力的补丁，大幅降低了对顶级模型端侧推理能力的依赖。
