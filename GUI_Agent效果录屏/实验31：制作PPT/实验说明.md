## 解决大文件上传问题

如果我有一个大文件（比如PPT），一开始是90MB。多次对该PPT进行修改、commit、push。逐渐达到了100+MB。这面临两个问题。
- 我每次成功的提交和push，都要上传数十MB的内容
- 当超过 100 MB 时，GitHub 不允许再上传

gemini的回答：
关于上传几十 MB 的内容：
- 虽然每次修改 PPT 还是需要上传整个文件（因为 PPTX 本质是压缩包，微小改动会导致二进制流巨大变化），但 LFS 优化了传输协议。
- 更重要的是，你的 .git 本地仓库不会再因为版本更迭而无限膨胀。普通 Clone 仓库的人只会下载当前版本的 PPT，而不会下载历史上所有的 PPT 版本，这大大节省了带宽和空间。

即使你现在开启 LFS，普通的 git push 依然会报错，因为那个超大的文件已经存在于你的本地 Git 历史记录中了。
你需要使用 LFS 的重写历史功能，把旧提交中的大文件“迁移”到 LFS 中：
```bash
# 将历史记录中所有的 pptx 文件转换为 LFS 存储
git lfs migrate import --include="*.pptx"
```

完整解决方案：
```
# 0. 确认当前状态是干净的
git status

# 1. 初始化 LFS 环境
git lfs install

# 2. 修改历史记录：将所有历史提交中的 pptx 转为 LFS 管理
# (执行后你会发现 Commit ID 变了)
git lfs migrate import --include="*.pptx" --everything

# 3. 彻底清理本地缓存空间（减小 .git 文件夹体积）
git reflog expire --expire=now --all
git gc --prune=now --aggressive

# 4. 强制推送到远程仓库（解决 GitHub 报错的关键）
# 假设你的分支名是 main，如果不是请替换
git push origin main --force
```
